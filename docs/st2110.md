# SMPTE ST 2110

bilbycast-edge ships first-class support for the broadcast-audio,
broadcast-data, and uncompressed-video subsets of SMPTE ST 2110:

| Standard | Essence | Status |
|----------|---------|--------|
| ST 2110-30 | Linear PCM audio (L16, L24) | Phase 1 |
| ST 2110-31 | AES3 transparent (Dolby E, etc.) | Phase 1 |
| ST 2110-40 | Ancillary data (RFC 8331) | Phase 1 |
| ST 2110-20 | Uncompressed video (RFC 4175, YCbCr-4:2:2 8/10-bit) | Phase 2 |
| ST 2110-23 | Single video essence across N -20 sub-streams (2SI, sample-row) | Phase 2 |
| ST 2110-22 | JPEG XS video | Deferred (pending libjxs) |

The accompanying NMOS surface area covers IS-04 (Node API), IS-05
(Connection Management), IS-08 (Audio Channel Mapping), and BCP-004
(Receiver Capabilities). See `nmos.md` for the protocol-level details.

## Architecture

ST 2110 essence flows reuse the existing `engine::flow` data plane: an
input task receives RTP packets, validates them with the essence-specific
depacketizer in `engine::st2110::*`, publishes whole `RtpPacket`s to the
flow's broadcast channel, and the output tasks resend them with optional
DSCP marking and Red/Blue duplication for SMPTE 2022-7 hitless
redundancy. There is no per-sample re-packetization on the loopback
path; a paired ST 2110 input + output achieves byte-identical loopback
by passthrough.

```
ST 2110-30 input  ──┐
                    │   broadcast::Sender<RtpPacket>
ST 2110-31 input  ──┼──►──────────────────────────────► ST 2110-30 output
                    │                                   ST 2110-31 output
ST 2110-40 input  ──┘                                   ST 2110-40 output
                                                        SRT output (WAN)
                                                        UDP output
```

### Pure-Rust constraint

The default build pulls in zero C dependencies. Run
`testbed/check-binary-purity.sh` after a release build to confirm `ldd`
shows only the libc/system libraries on the allow-list. The optional
`ptp-internal` feature (in-process `statime` PTP slave clock) is the only
non-default opt-in and is also pure Rust.

## PTP integration

bilbycast-edge does not run a PTP daemon itself in the default build.
The recommended deployment runs `ptp4l` from linuxptp on the host with a
hardware-timestamping-capable NIC; the edge polls the daemon's
management Unix socket once per second and reports the result through
the existing stats channel as `FlowStats.ptp_state`. Polling is
best-effort: when the daemon is unreachable the lock state stays at
`unavailable` and flows continue to operate.

```
ptp4l ──► /var/run/ptp4l ──► PtpStateReporter ──► PtpStateHandle ──► FlowStats.ptp_state
                                                                   │
                                                                   └─► manager UI
```

The optional `--features ptp-internal` flag is reserved for an in-process
`statime` slave clock for environments that prefer a single-process
deployment. The flag exists today as a build-matrix marker; the actual
`statime` integration lands in a follow-up.

### NIC requirements

For broadcast-grade lock (sub-µs offset, traceable holdover) you need a
NIC with hardware timestamping. Verified families:

- Intel i210, i350, X710, E810
- Mellanox ConnectX-5, ConnectX-6, ConnectX-7

Verify with `ethtool -T <iface>` — you want `hardware-transmit` and
`hardware-receive` in the timestamping capabilities.

### Host kernel tuning (receive side)

ST 2110-20 uncompressed video arrives at 1.6 Gbps (1080p50 8-bit) to
8.3 Gbps (2160p50 10-bit) per leg. The ingest requests a 64 MB
`SO_RCVBUF` per leg, but the kernel clamps every request to
`net.core.rmem_max` — distro defaults (4 MB, ≈8 ms of 4K traffic
effective) overflow into kernel drops on any scheduler hiccup or
sender burst, which surface as partial-frame drops (`packets_filtered`)
and decode artefacts downstream. Raise the ceiling persistently:

```bash
echo 'net.core.rmem_max = 67108864' | sudo tee /etc/sysctl.d/90-bilbycast-edge.conf
sudo sysctl --system
```

`packaging/install-edge.sh` installs this drop-in automatically. The
edge logs a `SO_RCVBUF … raise net.core.rmem_max` warning at input
start whenever the kernel clamps its request. Jumbo payloads
(`payload_budget` > ~1400) additionally need MTU 9000 end-to-end.

## SMPTE 2022-7 Red/Blue dual networks

Every ST 2110 input variant accepts an optional `redundancy` block that
points at a second bind address. When set, the input opens both Red and
Blue UDP sockets, runs the existing `HitlessMerger` over the merged
packet stream, and exposes per-leg counters via `FlowStats.network_legs`
(populated only when redundancy is configured). Outputs do the same in
reverse: every ST 2110 output accepts a redundancy block that
duplicates each packet to a second destination.

```
                  ┌── Red socket ──┐
RTP source ───►   │                │ ──► HitlessMerger ──► broadcast channel
                  └── Blue socket ─┘
```

Source-IP allow-listing is supported on the single-leg path. Combining
`allowed_sources` with `redundancy` is rejected by validation — the
merger path doesn't expose per-packet `src` and we don't want a silent
filter bypass.

| Field | Allowed values |
|-------|----------------|

## Per-output PCM transcoding (Audio Gateway phase)

Earlier ST 2110-30/-31 outputs were strict byte-identical RTP passthrough.
Every audio output now accepts an optional `transcode` block that runs an
in-line per-output PCM stage between the broadcast subscriber and the send
loop:

```json
"transcode": {
  "sample_rate": 44100, "bit_depth": 16, "channels": 2,
  "channel_map_preset": "5_1_to_stereo_bs775",
  "packet_time_us": 4000, "payload_type": 96,
  "src_quality": "high", "dither": "tpdf"
}
```

| Field | Allowed values | Default |
|---|---|---|
| `sample_rate` | 32000, 44100, 48000, 88200, 96000 | input rate |
| `bit_depth` | 16, 20, 24 | input depth |
| `channels` | 1..=16 | input count |
| `channel_map_preset` | `mono_to_stereo`, `stereo_to_mono_3db`, `stereo_to_mono_6db`, `5_1_to_stereo_bs775`, `7_1_to_stereo_bs775`, `4ch_to_stereo_lt_rt` | none |
| `packet_time_us` | 125, 250, 333, 500, 1000, 4000 | 1000 |
| `payload_type` | 96..=127 | 97 |
| `src_quality` | `high` (sinc), `fast` (polynomial) | `high` |
| `dither` | `tpdf`, `none` | `tpdf` |

The stage is **lock-free on the hot path** — `rubato` is the only new
dependency, all counters are `AtomicU64`, and IS-08 channel-map updates
propagate via a `tokio::sync::watch` channel without restarting the flow
(`engine::audio_transcode::MatrixSource::Is08Tracked` watches the active
map and refreshes its cached per-output routing matrix on every IS-08
activation via `watch::Receiver::has_changed()`). When `transcode` is
absent, the existing byte-identical passthrough path runs unchanged.

## Generic `rtp_audio` input/output

`InputConfig::RtpAudio` and `OutputConfig::RtpAudio` are RFC 3551 PCM-over-RTP
variants with the same wire format as ST 2110-30 but with relaxed
constraints — sample rates 32 / 44.1 / 48 / 88.2 / 96 kHz, no PTP, no
RFC 7273 timing reference, no NMOS clock_domain. They share the
`transcode` block and reuse the shared ST 2110 audio runtime with
`clock_domain` forced off. Use them for radio contribution feeds, talkback
between studios that don't share a PTP fabric, and ffmpeg / OBS interop.

## SMPTE 302M LPCM over MPEG-TS (SRT / UDP / RTP-MP2T)

Three audio output transports now support carrying uncompressed PCM as
**SMPTE 302M LPCM in MPEG-TS** for interoperability with the standard
broadcast tool stack:

| Output | `transport_mode` | Wire format |
|---|---|---|
| `srt` | `audio_302m` | 7×188 byte MPEG-TS chunks over an SRT live socket |
| `udp` | `audio_302m` | 7×188 byte MPEG-TS chunks as plain UDP datagrams |
| `rtp_audio` | `audio_302m` | 7×188 byte MPEG-TS chunks wrapped in RFC 2250 RTP/MP2T (PT 33) |

The pipeline is `Transcode (→ 48 kHz, even-channel, 16/20/24-bit) →
S302mPacketizer → TsMuxer (BSSD registration descriptor in PMT) →
S302mOutputPipeline (1316-byte chunk bundling)`. The 302M bit packing
follows ffmpeg's `libavcodec/s302menc.c` byte-for-byte; the round trip
between `S302mPacketizer::packetize_f32` and `S302mDepacketizer::parse`
is sample-exact at 16-bit and within 1 LSB at 24-bit (verified by 11
unit tests in `engine::audio_302m::tests`).

The `audio_302m` mode is mutually exclusive with FEC (`packet_filter`),
program filtering (`program_number`), and SMPTE 2022-7 redundancy.
Validation rejects all three combinations at config-load time.

Four runnable shell scripts in `testbed/audio-tests/302m-interop/`
exercise the wire boundary against `ffmpeg -c:a s302m`, `srt-live-transmit`,
and `ffprobe`. Each script auto-detects whether the required tools are
installed and prints `SKIPPED` if any are missing.

## Flow group runtime

`FlowManager::start_flow_group` and `FlowManager::stop_flow_group` provide
all-or-nothing group lifecycle management. `start_flow_group` spawns every
member flow's `FlowRuntime::start` in parallel via `futures::future::join_all`;
if any member fails, every member that did start is rolled back via
`destroy_flow` and an error is returned. Both are exposed as new manager
protocol commands (`start_flow_group` / `stop_flow_group`) — older edges
that don't recognize them fall through to the existing `Unknown command`
arm, so the protocol version is **not** bumped.

## Phase 2 — ST 2110-20 & ST 2110-23 video

Phase 2 adds uncompressed video ingest and egress. ST 2110-22 (JPEG XS) is
still deferred pending a libjxs wrapper crate.

### Scope
- **ST 2110-20**: RFC 4175 packetizer + depacketizer for YCbCr-4:2:2 at 8-bit
  and 10-bit. Other pgroup formats are validated-and-rejected.
- **ST 2110-23**: one video essence split across N ST 2110-20 sub-streams.
  `two_sample_interleave` and `sample_row` partition modes are supported;
  `sample_column` is validated-and-rejected.
- **2022-7 Red/Blue** per sub-stream (reuses `RedBluePair`).

### Format support & verified limits

**Scan: progressive only.** Interlaced video (1080i and friends) is not
supported as an ST 2110-20 essence in either direction. The RFC 4175
F-bit is parsed on ingest and written on egress, but the data model is
progressive-frame-only — no `Field1`/`Field2` frames are ever emitted,
no interlace config field exists, and the wire always signals F=0.
An interlaced *compressed* source (e.g. 1080i50 H.264) fed to a
2110-20 output is carried as **weaved progressive frames at the frame
rate** (25 fps for 1080i50) — verified live end-to-end with zero loss
— but a field-rate 2110 receiver will see "1080p25", not true 1080i.
Configure the output as `1920×1080` at the *frame* rate (25/30), not
the field rate.

**Sampling**: YCbCr-4:2:2 at 8-bit (`yuv422_8bit`, pgroup 4) and
10-bit (`yuv422_10bit`, pgroup 5). 4:2:0 / 4:4:4 / 12-bit / RGB have
no code path.

**Resolution**: 64–8192 px, both dimensions even. **Frame rate**:
1–240 fps via `frame_rate_num` / `frame_rate_den`, fractional rates
(`60000/1001`) included. The configured rate drives the ST 2110-21
raster pacer, so the **source frame rate must match the output's
configured rate** — a mismatch (e.g. 25 fps content into a 50 fps
output) warps the RTP/PES timeline downstream and is not a supported
configuration.

Live-verified matrix (2026-06-12, single 24-core host + Intel iGPU +
25 GbE, TX edge + RX edge on the same box, `hw_decode: "cpu"` on the
egress, RX ingest encode `h264_auto`/`hevc_auto` → QSV):

| Format | Wire rate | Result |
|--------|-----------|--------|
| 1080p50 4:2:2 8-bit | 1.68 Gbps | full rate, 0 drops, 0 loss, encode in==out 50.00 fps |
| 1080p59.94 8-bit | 2.02 Gbps | full rate (59.93 fps measured vs 59.940 nominal) |
| 1080p60 8-bit | 2.02 Gbps | full rate, 0 drops, 0 loss |
| 1080i50 (H.264 source → weaved 25p) | 0.84 Gbps | clean as 25p; no field signaling |
| 2160p50 4:2:2 10-bit | 8.32 Gbps | full rate, 0 drops, 0 loss |
| 2160p60 4:2:2 10-bit | 9.98 Gbps | full rate, 0 drops, 0 loss |

CPU egress decode is content-dependent: clean/synthetic HEVC decodes
2160p60 at rate on this host class, while a dense broadcast UHD HDR
capture topped out ≈45 fps at 2160p50. HW egress decode (`hw_decode`
unset = `auto`, the default) is the escape hatch for such sources.

**HW egress decode + the raster schedule.** The ST 2110-21 raster
schedule anchors at the **first decoded frame** (absorbing demux
warm-up plus the HW decoder's ~0.5 s open and pipeline depth),
re-anchors when frames run persistently behind their slots (source
stall recovery, source-vs-TAI rate drift — a *draining* backlog from a
delivery clump rides through losslessly instead), and the sender paces
packetization to the schedule and waits for wire-ring capacity so
decode bursts can never tail-drop packets mid-frame (RFC 4175
receivers discard partial frames). Bursts beyond the ~9-decoded-frame
channel cushion shed as clean whole-frame skips, logged rate-limited.
For `auto`, the egress decode resolver prefers **NVDEC ≻ QSV ≻ VAAPI ≻
CPU** — unlike the display path, this path needs sysmem frames, and
VAAPI's hwframe download measured 46 fps at 2160p50 10-bit on Intel
iGPU where QSV sustains the full 50 (2026-06-12; 1080p-class rasters
run at rate on every backend). Live-verified with `hw_decode: auto`:
1080p50 (1.68 Gbps) and 2160p50 10-bit (8.32 Gbps), both full rate,
0 drops, 0 loss, `pcr_trust` window p95 ≤ 2.3 ms.

**PMT `stream_type` vs the actual ES codec.** The flow assembly's
declared `stream_type` (27 = H.264, 36 = HEVC) drives the egress
decoder, and two layers now keep it truthful instead of silently
killing video:

- **Assembler reconciliation** — every PSI tick the assembler compares
  each video slot's declared `stream_type` against the live source PMT
  (debounced, video family only, Switch slots excluded). On a
  confirmed mismatch it corrects the assembled PMT (version bump, so
  every downstream demuxer/decoder re-locks without a flow restart)
  and raises a Warning `pid_bus_stream_type_mismatch` event with
  `{declared, observed, program, out_pid, source}` — the config still
  declares the stale byte, so fix the assembly to clear the warning.
- **Egress bitstream sniffer** — the 2110-20/-23 decode worker
  independently classifies the NAL headers; on a confirmed
  PMT-vs-bitstream contradiction it decodes with the detected codec
  and raises `st2110_video_codec_mismatch`. This also covers live
  source codec changes that arrive with an unchanged PMT version
  (verified: x264 → x265 source swap mid-flow recovers to full wire
  rate within ~1 s). A keyframe-gated stall watchdog
  (`st2110_egress_decode_stalled`) backstops undecidable bitstreams by
  retrying the software decoder, then the alternate codec.

**A/V sync across a 2110-20 + 2110-30 pair.** Both essences now carry
the **source PES timeline** on their RTP timestamps (the -30
compressed-audio bridge derives its timestamps from the source PTS,
continuity-bridged exactly like the -20 video wire timestamps), and a
receiving flow anchors every 2110 input's synthesised PES timeline on
one shared per-flow origin (`engine::st2110::timeline`), resolving
each essence's first wire timestamp across both RTP moduli and the
2³³ PES wrap. AAC encode hops also label output frames content-true
(the fdk encoder's codec delay, ≈2600 samples, is subtracted at the
PTS anchor). Net effect, measured with the trusted flash+beep source
through SRT → flow → 2110-20 + 2110-30 → second edge → re-encode →
SRT: **the chain adds 0.0 ms of A/V offset with 0.000 ms/s drift**
(2026-06-12; the pre-fix chain added +56 ms of a measured +160 ms —
the remainder was the test source's own intrinsic offset, measured
identically on a direct source capture). Senders whose essences use
unrelated RFC 3550 RTP epochs fall back to arrival anchoring with a
warning — cross-essence offset then reflects path-latency differences,
which the per-flow lipsync trim (`set_master_clock_lipsync`, ±200 ms)
can dial out.

**Signaling**: SDP generation and NMOS IS-04/IS-05 exposure are
implemented for -30/-31/-40 only; ST 2110-20/-23 senders/receivers are
configured manually and do not yet appear in NMOS or emit SDP.

### Data path (non-blocking)

```
INPUT (ST 2110-20 / -23)
  UDP ─► RedBluePair ─► Rfc4175Depacketizer ─► mpsc(raw frames)
                                                  │
                           spawn_blocking: VideoEncoder + TsMuxer
                                                  │
                                                  ▼
                                 broadcast_tx (RtpPacket, is_raw_ts = true)

OUTPUT (ST 2110-20 / -23)
  broadcast_rx ─► TsDemuxer ─► mpsc(H.264/HEVC access units)
                                                  │
                        spawn_blocking: VideoDecoder + VideoScaler
                                                  │
                              pack to pgroup bytes → mpsc(raw frames)
                                                  │
                              Rfc4175Packetizer ─► UDP Red (+ Blue)
```

All codec work runs inside `tokio::task::spawn_blocking`. Bounded mpsc
channels drop on lag so the tokio reactor is never blocked.

### Licensing

ST 2110-20 / -23 **inputs** require one of `video-encoder-x264`,
`video-encoder-x265`, `video-encoder-nvenc`, or `video-encoder-qsv`
Cargo features at build time (or the `video-encoders-full` composite
used by the `*-x86_64-linux-full` release artefact, which bundles
all four; the `*-aarch64-linux-full` artefact omits QSV because
Intel iGPU is x86_64-only). The default `cargo build` has no
software video encoders, so it cannot drive -20 / -23 inputs —
validation would accept the config but the encode worker will log
an error and drop frames. ST 2110-20 / -23 **outputs** only need
`media-codecs` (default on).

- ST 2110-22 (JPEG XS).
- Interlaced essences (1080i as true field-rate 2110-20 with F-bit

## ST 2110-21 narrow profile pacing (`SO_TXTIME` + ETF)

The default ST 2110-20 / -23 egress path is unpaced — the kernel
schedules transmission as fast as the NIC will accept it, which on
narrow-profile receivers (Imagine, Lawo, EVS Xeebra, Grass Valley LDX,
Bridge Tech VB330 in test mode) breaks the VRX bound the receiver
expects per ST 2110-21 §6.3. For broadcast-grade interop with those
receivers, opt the output into kernel-paced wire emission via
`SO_TXTIME` + the Linux ETF qdisc.

### Architecture

`engine::wire_emit_txtime` is a Linux-only `libc::sendmsg` /
`sendmmsg` wrapper that attaches an `SCM_TXTIME` CMSG to each
datagram. `engine::st2110::pacer::St2110_21Pacer` computes the
per-packet `target_tx_time_ns` against the frame raster, anchored on
PTP `CLOCK_TAI` (re-anchors when the PTP daemon emits a fresh sample;
falls back to monotonic if PTP isn't running). The two compose in
`engine::st2110_video_io::run_paced_sender` — a `spawn_blocking`
thread that owns the `std::net::UdpSocket`, runs the pacer math, and
ships per-packet timestamps into the kernel's ETF qdisc.

Full reference: [`docs/wire-pacing.md`](wire-pacing.md) Phase 2.

### Opt-in

```json
{
  "type": "st2110_20",
  "id": "video-out-1",
  "wire_pacing": {
    "mode": "tx_time",
    "profile": "narrow"
  },
  ...
}
```

`profile` is one of `narrow` (default), `narrow_linear`, `wide` —
v1 of the pacer treats all three as `narrow_linear` (even pacing
across the active video period). Classic gapped narrow is a follow-up
when a real receiver requires it.

The capability `wire_pacing_txtime` is advertised on
`HealthPayload.capabilities` only when the host successfully sets
`SO_TXTIME` on a probe socket at startup (`engine::hardware_probe`).
Manager UI gates the per-output dropdown on this capability — older
edges, non-Linux hosts, kernels < 4.19, and containers without
permission all hide the option automatically.

### Operator runbook

The edge does NOT install the ETF qdisc itself (qdisc installation
needs `CAP_NET_ADMIN`, deliberately operator-side). After deploy:

```bash
sudo bash packaging/setup-etf-qdisc.sh enp1s0   # name your egress NIC
```

The script installs `mqprio` at root (3 traffic classes, one per
hardware tx queue) and `etf clockid CLOCK_TAI delta 200000 offload`
on the prioritized class. `offload` enables NIC HW pacing on
supported NICs; falls back to software ETF on unsupported ones (still
~1–10 µs jitter, an order of magnitude better than no pacing).

Tested NIC families with HW offload:

- Mellanox CX-6 / CX-7 (mlx5_core)
- Intel E810 (ice driver)
- Intel i210 (igb driver)

Removal: `sudo tc qdisc del dev <if> root`.

PTP prerequisite: `ptp4l` + `phc2sys` running, system clock
disciplined to a grandmaster on the production fabric. ST 2110-21
narrow profile is meaningless without a common PTP reference between
sender and receiver.

### Failure modes

| What | Behavior |
|---|---|
| Kernel < 4.19 or container without permission | startup probe fails; capability not advertised; output configs that opted in still spawn but fall back to plain `send_to`. Operator sees a `wire_pacing_unavailable`-style WARN log. |
| ETF qdisc not installed | `setsockopt(SO_TXTIME)` succeeds but the kernel queue is `pfifo_fast` — no actual pacing. Same observable behavior as today's unpaced path; no regression. Detect via `tc -s qdisc show dev <if>`. |
| `ptp4l` not running | pacer's frame epoch never re-anchors after the startup snap. Falls back to monotonic with drift; tracks loosely against absolute time. Receivers expecting ST 2110-21 narrow may flag VRX overflow. |
| Burst landed in past (`EOVERFLOW`) | datagram dropped; counted under `OutputStats.packets_dropped`. The `OutputStats.wire_pacing_late` counter exists and is populated from the SO_TXTIME errqueue drain on the PCR/raster-paced UDP/RTP/302M/ST 2110 outputs, but the ST 2110-21 txtime path does not yet increment it — these late drops still roll into the existing drop counter for now. |

### Verification

Receiver-grade narrow-profile compliance is operator-side — automated
end-to-end CI cannot verify it without a hardware ST 2110 analyzer.
Manual checklist for production validation:

1. **Capability check:** `curl -k https://<edge>:8080/health | jq .capabilities`
   should include `"wire_pacing_txtime"`.
2. **qdisc check:** `sudo tc -s qdisc show dev <if>` should list `mqprio`
   and `etf` (not `pfifo_fast`).
3. **Inter-packet pacing:** with the qdisc installed, capture the
   stream with `tcpdump -i <if> -ttt 'udp and host <dst>'` and check
   inter-packet times. With ETF + PTP-disciplined NIC: even pacing
   within a few µs of `frame_period_ns / packets_per_frame`. Without
   ETF: bursts at NIC speed.
4. **Compliance analysis** (Bridge Tech VB330, EBU LIST, Telestream
   PRISM): plug the analyzer onto the egress LAN and confirm narrow
   profile bounds. v1's `narrow_linear` pacing should pass narrow
   bounds on most receivers; if a specific receiver complains, file
   a follow-up to add gapped-narrow mode.
